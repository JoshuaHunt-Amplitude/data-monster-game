Speedrunning software

How Amplitude 3×'d its PRs by turning the software factory into a speedrun — and moved engineers from in the loop to on it.

Have you ever tried to speedrun a video game? Most people think it takes faster reflexes — do the same thing every time, just mechanically faster. It's not that simple. Speedrunners spend most of their time studying the route and engineering the system and environment to go faster each time.

Building our autonomous software factory feels the same. We wanted to see how fast we could get software into the hands of our customers.

After X months, we've 3×'d our PRs.

Why speed matters

Once the factory is fast, any idea becomes possible

AI has made building cheap. Knowing what to build and how it works is the new game. — Wade Chambers, Chief Engineering Officer, Amplitude

Once the software factory gets fast, any idea becomes possible and the real judgment is in knowing which ones are worth trying. You can build the smallest version, put it out there, see if it moved your guiding metric, then do it all over again.

The goal is to run smaller experiments and see good ideas surface much faster. With the same headcount, that's how you out-iterate everyone else.

By speeding up our software factory, we're moving engineers from in the software loop, doing every step, to on the loop, steering it.

The constraint

The gates slowing us down

Building software is untidy, but the path it goes through is less so. An idea becomes code, the code gets validated and reviewed, then it gets deployed to production. For us, that path slowed down at a few specific steps.

The pipeline · gates in gold
Before our bet, every step was manual except staging deploys. The heaviest gates that stayed manual: validation, review, and approval.

Despite its flaws, this design worked for years because writing code was the slowest part of the system — a natural limit on how many changes could enter the pipeline at once. But agents changed that. Faster code generation meant a trickle of PRs suddenly became a flood, and the manual validating, reviewing, and approving became the stoppage points.

This is the Theory of Constraints: speeding up one part of the system just moves the goal post to the next-slowest step. What a beautifully messy problem to solve.

Rebuilding the foundations

Fix how we ship, not just what

When Stripe built their coding agent, they found something interesting: what's good for agents is good for humans, too. We agree — but we hadn't made it a priority. For years we optimized for what we shipped; agents forced us to finally fix how we ship.

Local app start

One containerized environment

We replaced local setups with a single containerized definition that anyone — human or agent — can bring up with one command. The frontend is fully containerized; the backend is mid-migration. The bigger next step is making those environments ephemeral, so full-stack changes can be validated and agents can run in parallel without colliding.

CI from 25 minutes to under 5

Frontend CI runs the checks every PR needs — formatting, linting, type-checking, build, and unit tests — in under five minutes, at roughly a tenth of the cost. When CI checks are this fast, most engineers stopped coding on their machines and just started pushing PRs.

Automated legacy migrations

We pointed agents at the patterns that confused them most and let them run the migrations themselves. The largest: three competing design systems in one codebase, consolidated onto core-ui. We codified the changes as agent rules and skills backed by lint checks, so agents and engineers don't drift back into old patterns. These were migrations we'd written off for years.

How we develop now

From agent-assisted to agent-written

Improving our DevEx changed how people code in expected and unexpected ways. The obvious: agents write a lot of code now. More surprising: where that code gets written.

Agents took the keyboard

Like most orgs, our engineers shifted from writing code by hand to steering Cursor and Claude Code agents. Progress is uneven — the frontend is far along while the backend lags, because agents can't yet validate those changes safely enough to merge with confidence. In general, agent-written code roughly tracks how old the repo is.

CI as our new dev environment

Once frontend CI dropped under 5 minutes, working locally was no longer the fast option. Engineers push a branch, let CI run, and review in GitHub. Because it runs in the cloud, they aren't limited to one change at a time — they can kick off 20 PRs before logging off and review them the next morning. A migration might hit 100 PRs while feature work stays around 1–3.

Gate 1

Validation

Validation is where we confirm a change does what it's supposed to. In theory every change should go through it. In practice ours was slow and the first thing to go under tight deadlines — especially for new features, where it felt safer to ship without testing something no one was using yet. The result: features launched with unknown bugs customers found later, after the team had lost context and moved on.

The code we skip testing today becomes the legacy code of tomorrow. We had to make validation fast enough not to skip: ship a PR, get a working preview, no local setup.

Frontend previews

Built assets go up to S3 and load as a working copy of the app, with dev tools to flip flags and a deep link straight to the page your change affects. Checking a change takes 2–5 minutes of click-through, no local setup.

Backend ephemerals

Backend changes get the same treatment: a live, throwaway copy of the infrastructure spun up for the PR, with a link to hit the APIs and see what actually works before anything merges.

Gate 2

Review and approval

In the old world every PR needed a human to approve it, but our review culture defaulted to rubber-stamping. Fast, but shallow — and it created social friction anytime someone did give thorough feedback, because it wasn't the norm.

An arbitrary review gate that added time to every change while catching only a fraction of the bugs didn't make sense. So we removed it. Sort of. An automated reviewer now runs on every PR and auto-approves low-risk changes based on our risk model.

Automated code review

We run Cursor's Bugbot on every PR in CI. It works out of the box, and we add context when it misses things. So far it catches bugs most humans used to let through.

Risk-based auto-approval

A risk model scores each change on size, scope, what it touches, test coverage, and whether it modifies public APIs or data models. Low-risk changes with resolved comments merge on their own. When someone catches a change that shouldn't have merged, it goes into an eval suite and we tune the model. And yes — it passes SOC2: the audit doesn't need a human to click approve, it needs documented criteria, logged decisions, and a way to override.

The hard part

A culture of AI transformation

We improved our systems and tooling and got far, fast. The harder part was creating the culture to get there.

AI progression is largely not a technical issue. It's a cultural one. — Wade Chambers, Chief Engineering Officer, Amplitude

Start with the leaders

You can't ask engineers to hand their work to an agent if their own leaders won't. So we made them go first, literally — our CEO of engineering asked senior leaders to vibe code in front of the entire EPD org, live, highs and lows and all. They took the exercise back to their own teams, co-developing rather than talking at them.

Let anyone ship

The biggest positive was handing over the keys and letting non-engineers ship code. Designers, PMs, even marketers use Claude and Cursor to fix bugs and make frontend improvements — faster, because there's no handoff, and often better, because a designer sees the UI with a clear eye. They're ~5% of our PRs today, but high-value, and growing.

Spend where it counts

We don't cap AI spend — if anything we encourage more. But the model you pick matters as much as how you run it. Our CTO recently refactored a major backend to have proper typings for $500 using a cheaper model and a well-planned script; the same work could have easily cost tens of thousands. Don't feel constrained by tokens — choose the right model for the job.

It's never perfect

We made mistakes. First, spreading the effort too thinly across the org — the peanut-butter approach — before shifting to a tiger-team model with AI-pilled engineers embedded across teams. Second, trying to build agents before we had the infrastructure. There was no need to jump to the end: prioritizing foundational DevEx is what ultimately helped us move faster.
