Have you ever tried to speedrun a video game?

Most people think it takes superhuman reflexes — do the same thing every time, just mechanically faster. It's not that simple. Speedrunners spend most of their time studying the route and engineering the system and environment to go faster each time.

Building our autonomous software factory feels the same. We wanted to see how fast we could get software into the hands of our customers.

After 6 months, we've 3×'d our PRs — and this is how we did it, so you can too.

Why speed matters

Why does speed matter, really?

AI has made building cheap. Knowing what to build and how it works is the new game. — Wade Chambers, Chief Engineering Officer, Amplitude

Once the software factory gets fast, any idea becomes possible and the real judgment is in knowing which ones are worth trying. You can build the smallest version, put it out there, see if it moved your guiding metric, then do it all over again.

The goal is to run smaller experiments and see good ideas surface much faster. With the same headcount, that's how you out-iterate everyone else.

By speeding up our software factory, we're moving engineers from in the software loop — doing every step — to on the loop, steering it.

The constraint

The gates slowing us down

Building software is untidy, but the path it goes through is less so. An idea becomes code, the code gets validated and reviewed, then it gets deployed to production. For us, that path slowed down at a few specific steps.

The pipeline · gates in gold
Before our bet, every step was manual except staging deploys. Engineers pieced together unreproducible local environments that could take up to 30 minutes to spin up — on a bad day, most of their time went to setting up and shipping a single change.

Despite its flaws, this design worked for years because writing code was the slowest part of the system — a natural limit on how many changes could enter the pipeline at once. But agents changed that. Faster code generation turned a trickle of PRs into a flood, and the manual validating, reviewing, and approving became the stoppage points.

This is the Theory of Constraints: speed up one part of the system and the goal post just moves to the next-slowest step. A beautifully messy problem — and we were up for it. The heaviest gates to push through were the ones that stayed manual: validation, review, and approval.

Rebuilding the foundations

Fix how we ship, not just what

When Stripe built their coding agent, they found something interesting: what's good for agents is good for humans, too. We agree — but we hadn't made it a priority. For years we optimized for what we shipped; agents forced us to finally fix how we ship. The bright side? It also made us improve much faster than expected.

One containerized environment

We replaced local setups with a single containerized definition that anyone — human or agent — can bring up with one command. It spins up in seconds now, down from 25–35 minutes. The frontend is fully containerized and the backend is mid-migration; the bigger next step is making those environments ephemeral, so full-stack changes can be validated and agents can run in parallel without colliding.

CI from 25 minutes to under 5

Frontend CI runs the checks every PR needs — formatting, linting, type-checking, build, and unit tests — in under five minutes, at roughly a tenth of the cost. Our code lives in a Turborepo monorepo, so remote build caching and running only the projects a change touches did most of the work. When CI is this fast, most engineers stopped coding on their machines and just started pushing PRs.

Automated legacy migrations

We pointed agents at the patterns that confused them most and let them run the migrations themselves. The largest was three competing design systems in one codebase — Ant Design, in-house Cerulean, and core-ui (Radix + CVA + Tailwind) — consolidating onto core-ui. We did the same with i18n, three translation systems down to one, then codified the changes as agent rules and skills backed by lint checks so no one drifts back into old patterns. Migrations we'd written off for years, finally possible.

How we develop now

From agent-assisted to agent-written

Improving our DevEx changed how people code in expected and unexpected ways. The obvious: agents write a lot of code now. More surprising: where that code gets written.

Agents took the keyboard

Like most orgs, our engineers shifted from writing code by hand to steering Cursor and Claude Code agents — but the progress is uneven. The frontend is far along, while the backend lags because agents can't yet validate those changes safely enough for a human to merge with confidence. Nova, our oldest repo, is still largely hand-written. In general, agent-written code tracks with how old the repo is. The goal: increase agent-generated code across surfaces and eventually let agents take a spec and own the whole change.

CI as our new dev environment

Once frontend CI dropped under five minutes, working locally was no longer the fast option. Engineers push a branch, let CI run, and review it in GitHub — CI went from the gate you clear after building to the dev environment itself. Because it runs in the cloud, they aren't limited to one change at a time: they can kick off 20 PRs before logging off and review them the next morning. The only limit is knowing what to build, which is why a migration might hit 100 PRs while feature work stays around 1–3.

Gate 1

Validation

Validation is where we confirm a change does what it's intended to do. In theory every change should go through it. In practice ours was slow and the first thing to go under tight deadlines — especially for new features, where it felt safer to ship without testing something no one was using yet. The result: features launched with unknown bugs customers found later, after the team had lost context and moved on.

The code we skip testing today becomes the legacy code of tomorrow. We had to make validation fast enough not to skip. Automated tests help, but they don't catch what you'd only notice by trying the change yourself — so we made that easy: ship a PR, get a working preview, no local setup.

Frontend previews

For a frontend change, the built assets go up to S3 and load as a working copy of the app — with dev tools to flip feature flags and settings, and a deep verification link straight to the page your change affects. Checking a change takes 2–5 minutes of click-through, no local setup needed.

Backend ephemerals

Backend changes get the same treatment through ephemerals: a live, throwaway copy of the infrastructure spun up for the PR. You get a link to hit the APIs and see what actually works before anything merges.

Gate 2

Review and approval

In the old world every PR needed a human to approve it, but our review culture defaulted to rubber-stamping. Fast, sure — but shallow, and it created social friction anytime someone did give thorough feedback, because it wasn't the norm.

How did we get here? We'd adapted to a system that doesn't support meaningful review. Without strong automated validation signals from Gate 1, reviewers would have to manually verify behavioral correctness for every PR — unrealistic at any real throughput — so they skim and approve on trust and structure. An arbitrary gate that added time while catching a fraction of the bugs didn't make sense. So we removed it. Sort of. An automated reviewer now runs on every PR and auto-approves low-risk changes based on our risk model.

Automated code review

We run Cursor's Bugbot on every PR in CI. It works out of the box, and we add context when it misses things. So far it catches bugs most humans used to let through — even though we've tripled the PRs we ship, bug reports have fallen by more than half and high-priority issues have dropped too. More code hasn't meant worse code, and Bugbot is a big reason why.

715 → 319
Bug reports / month · ↓ 55%

Risk-based auto-approval

A risk model scores each change on size, scope, what part of the codebase it touches, test coverage, and whether it modifies public APIs or data models. If a change is low-risk and its comments are resolved, it merges on its own — no human in the loop. It's live for our JS apps and Amplitude, but not yet for Nova. When someone catches a change that shouldn't have merged, it goes into an eval suite and we tune the model so it doesn't happen again.

Does it pass SOC2?

Turns out SOC2 doesn't need a human to click approve — it needs demonstrable controls: documented criteria, logged decisions, and a way to override. So we documented all of it: what qualifies for auto-approval, what still needs a human, and how to escalate. Approval and validation are separately logged signals, so every decision can be audited after the fact. We brought compliance in early so we didn't have to redesign it later.

The hard part

A culture of AI transformation

We improved our systems and tooling and got far, fast. The harder part was creating the culture to get there. Engineers are craftspeople who built careers on clean abstractions, reusable interfaces, and code others can build on — the idea that an agent can do that just as well isn't easy to hear. Getting to broad adoption took two waves: AI fluency (moving from chat-based questions to goal-oriented tasks), then using that fluency to disrupt entire workflows.

AI progression is largely not a technical issue. It's a cultural one.— Wade Chambers, Chief Engineering Officer, Amplitude

Start with the leaders

Leader ambition can be one of the biggest drags on velocity. You can't ask engineers to hand their work to an agent if their own leaders won't — so we made them go first, literally. Our chief engineering officer asked senior leaders to vibe code in front of the entire EPD org, experiencing the highs and lows of agentic building live on stage. They took the exercise back to their teams, co-developing rather than talking at them.

Let anyone ship

The biggest positive was handing over the keys and letting non-engineers ship code. Designers, PMs, even marketers use Claude and Cursor to fix bugs and make frontend improvements — faster, because there's no handoff, and often better, because a designer sees the UI with a clear eye and a PM acts on real customer feedback. It was only possible because of the validation and review changes: preview links let them check a change without running anything locally, and each PR deep-links to the exact page that changed. Our advice: make it wildly easy — strong docs, first-time-setup training, and an internal marketplace of shared skills.

Spend where it counts

We don't cap AI spend — if anything, we encourage more. But the model you pick matters as much as how you run it, so we watch high bills and check whether a lighter model would work. A cheaper model lets you run two or three times as much on the same budget. Our CTO recently refactored a major backend to proper typings for $500 using a cheaper model and a well-planned local script; the same work could have easily cost tens of thousands. Don't feel constrained by tokens — choose the right model for the job.

It's never perfect

We made mistakes. First, spreading the effort too thinly across the org — the peanut-butter approach — before shifting to a tiger-team model with AI-pilled engineers embedded across teams. Second, trying to build agents before we had any of the infrastructure in place. There was no need to jump to the end: prioritizing our foundational DevEx is what ultimately helped us move faster.
