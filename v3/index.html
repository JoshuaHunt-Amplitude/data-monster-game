Have you ever tried to speedrun a video game?

Most people think it takes faster reflexes — do the same thing every time, just mechanically faster. It's not that simple. Speedrunners spend most of their time studying the route and engineering the system and environment to go faster each time.

Building our autonomous software factory feels the same. We wanted to see how fast we could get software into the hands of our customers.

After 6 months, we've 3×'d our PRs.

Why speed matters

Why does speed matter, really?

AI has made building cheap. Knowing what to build and how it works is the new game. — Wade Chambers, Chief Engineering Officer, Amplitude

Once the software factory gets fast, any idea becomes possible and the real judgment is in knowing which ones are worth trying. You can build the smallest version, put it out there, see if it moved your guiding metric, then do it all over again.

The goal is to run smaller experiments and see good ideas surface much faster. With the same headcount, that's how you out-iterate everyone else.

By speeding up our software factory, we're moving engineers from in the software loop, doing every step, to on the loop, steering it.

The constraint

The gates slowing us down

Building software is untidy, but the path it goes through is less so. An idea becomes code, the code gets validated and reviewed, then it gets deployed to production. For us, that path slowed down at a few specific steps.

The pipeline · gates in gold
Before our bet, every step was manual except staging deploys. The heaviest gates that stayed manual: validation, review, and approval.

This design worked for years because writing code was the slowest part of the system. But agents changed that — a trickle of PRs became a flood, and the manual steps became the stoppage points. This is the Theory of Constraints: speed up one part and the goal post moves to the next-slowest step.

One containerized environment

We replaced local setups with a single containerized definition anyone — human or agent — can bring up with one command. Next: making those environments ephemeral so agents can run in parallel without colliding.

CI from 25 minutes to under 5

Frontend CI runs every check a PR needs in under five minutes, at a tenth of the cost. When CI is this fast, most engineers stopped coding locally and just started pushing PRs.

Automated legacy migrations

We pointed agents at the patterns that confused them most and let them run the migrations — consolidating three design systems onto one. Migrations we'd written off for years, finally possible.

How we develop now

From agent-assisted to agent-written

Improving our DevEx changed how people code. The obvious: agents write a lot of code now. More surprising: where that code gets written.

Agents took the keyboard

Engineers shifted from writing code by hand to steering Cursor and Claude Code agents. The frontend is far along; the backend lags, because agents can't yet validate those changes safely enough to merge with confidence.

CI as our new dev environment

Once frontend CI dropped under 5 minutes, working locally was no longer the fast option. Because it runs in the cloud, engineers can kick off 20 PRs before logging off and review them the next morning.

Gate 1

Validation

Validation is where we confirm a change does what it's supposed to. Ours was slow and the first thing to go under deadlines — so features launched with unknown bugs customers found later. The code we skip testing today becomes the legacy code of tomorrow.

Frontend previews

Ship a PR, get a working preview — built assets on S3, dev tools to flip flags, and a deep link to the exact page your change affects. 2–5 minutes of click-through, no local setup.

Backend ephemerals

Backend changes get a live, throwaway copy of the infrastructure spun up for the PR, with a link to hit the APIs and see what works before anything merges.

Gate 2

Review and approval

Every PR needed a human to approve it, but our culture defaulted to rubber-stamping. An arbitrary gate that added time while catching a fraction of the bugs didn't make sense — so we removed it. Sort of. An automated reviewer now runs on every PR and auto-approves low-risk changes.

Automated code review

We run Cursor's Bugbot on every PR in CI. It catches bugs most humans used to let through.

Risk-based auto-approval

A risk model scores each change on size, scope, coverage, and whether it touches public APIs. Low-risk changes with resolved comments merge on their own — and it passes SOC2, which needs documented criteria and logged decisions, not a human clicking approve.

The hard part

A culture of AI transformation

AI progression is largely not a technical issue. It's a cultural one.— Wade Chambers, Chief Engineering Officer, Amplitude

Start with the leaders

You can't ask engineers to hand work to an agent if their leaders won't. So we made them go first — senior leaders vibe coded in front of the entire EPD org, live, highs and lows and all.

Let anyone ship

We handed non-engineers the keys. Designers, PMs, even marketers use Claude and Cursor to fix bugs and ship frontend improvements — faster, because there's no handoff.

Spend where it counts

We don't cap AI spend. But the model you pick matters as much as how you run it — our CTO refactored a major backend to proper typings for $500 with a cheaper model; the same work could have cost tens of thousands.
